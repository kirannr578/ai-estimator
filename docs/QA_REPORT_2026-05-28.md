## 4. Coverage gaps (consolidated)

Surfaces intentionally not exercised under the no-LLM-spend + test-only-changes guardrail. Group into "blocker-class" (need infra change) vs "scope-class" (can ride a follow-on slice):

### Blocker-class — need infra to cover

1. **Deep CLI flows** (`--client-pdf` end-to-end, `--render-proposal`). Today the only way to exercise them is to spend real LLM credits. **Recommendation:** introduce an `ESTIMATOR_LLM_MOCK_PROFILE=<name>` env-var hook on `LLMClient` that loads a deterministic stub from `tests/fixtures/llm_mocks/<name>.json`. One small slice, unlocks ~5 deep CLI / proposal scenarios. (YY-3 §5 items 1-2.)
2. **Real-world scanned PDFs.** Synthetic fixtures use `pymupdf.insert_text` (vector text); image-only scans take a different `is_scanned` code path that is unexercised here. (YY-1 §5.)
3. **CWICR real-matcher network/disk round-trip.** All confidence-pricing tests use a `_FakeMatcher`; the real `CwicrMatcher` is a TF-IDF + nearest-neighbour search against the bundled dataset. Recommend a single end-to-end smoke test against a hand-curated 10-row stub dataset. (YY-2 §5 item 1.)

### Scope-class — can ride a follow-on slice

4. **8 of 12 typed schedule extractors not directly QA-touched** (finish, panel, lighting, HVAC, plumbing, kitchen, lab casework, AV, security). They reuse the same `header_utils` helper and per-family shape so QA confidence transfers, but a future slice should hit at least one mechanical-trade and one specialty-trade family directly. (YY-1 §5.)
5. **xlsx parser cell-type coercion** for currency strings like `"$3,850.00"` vs the float `3850.00`. (YY-2 §5 item 2.)
6. **Multi-page sub-quote LLM-vision** with table continuation across page breaks. (YY-2 §5 item 3.)
7. **Mixed ADDITIVE + DEDUCTIVE + VE alternate selection-toggle rollup** — current tests pin ADDITIVE-only. (YY-2 §5 item 5.)
8. **`--no-drawings` content-classifier accuracy** — current implementation is a 5 MB size heuristic, not content-based. Worth a follow-up if drawings ever shrink below 5 MB on vector-only sets. (YY-3 §5 item 3.)
9. **Output-dir overwrite semantics on a successful run** — covered only for early-exit; blocked on the same LLM-mock-hook gap as (1). (YY-3 §5 item 5.)

## 8. Process retrospective — what worked, what to repeat

### Worked

* **Decomposition after `resource_exhausted`.** Predecessor Worker YY errored on the 10-subsystem brief. Splitting into 3 workers × 3-4 subsystems each (with disjoint file ownership) ran cleanly and in parallel — total wall-clock from dispatch to final synthesis was ~18 minutes.
* **Per-group report file isolation.** Each worker writes its OWN report file. Zero contention on `docs/QA_REPORT_*.md`. The top-level synthesis (this document) is owned by the coordinator after all three returned.
* **XFAIL-only-no-fix guardrail.** Bugs surface without bleeding into `core/*` edits during the QA pass itself. This kept the three workers from contending on `core/estimator.py` / `core/pricing/*` / `core/exporter*.py`. The companion PASS pin (locking today's contract) is the key engineering nuance — it means the fix-sprint will mechanically flip both tests together with high reviewer confidence.
* **Read-budget caps.** YY-1 hit 10/10, YY-2 hit 12/12, YY-3 used 8/11. Tight budgets force focus and prevent the runaway-reads pattern that probably contributed to the original `resource_exhausted`.
* **Bug-id prefixes** (`B1-N`, `B2-N`, `B3-N`) made consolidation in this synthesis trivial — no rename needed.

### To repeat

* For any future comprehensive sweep across the estimator pipeline, **default to a 3-way subsystem decomposition** (ingest/extraction · pricing/overrides · exports/CLI). The boundary cuts cleanly along file ownership.
* **Bound scenario density to ~7 per subsystem per worker.** YY-1 went a little high at ~15 per subsystem and finished cleanly only because the schedule-extractor scenarios were structurally similar. The 7-per-subsystem budget that YY-2 and YY-3 used is the sustainable target.
