## Template dogfood notes (2026-05-27)

> Surface every quirk hit while using `bids/_TEMPLATES/texas-state-csp-hsp/` to seed this workspace. These become improvement tickets for the template — keep here for Day-3 triage.

1. **No `00-overview.md`** — the template ships `01-overview.md` as the first file. Cross-workspace convention has been inconsistent (TAMU Wehner also uses `01-overview.md`; the user spec for Phase C asked for `00-overview.md`). **Recommendation:** either rename to `00-overview.md` or update the workspace-spec convention to reflect `01-`.
2. **No `01-scope.md`** — the template ships `04-scope-of-work.md` as the scope file; the user spec asked for `01-scope.md`. **Recommendation:** add a redirect / README pointer so users can find the scope file by either name.
3. **No `04-checklist.md`** — the template ships `02-bid-prep-checklist.md`. Same naming-convention drift as above.
4. **No `proposal/` subfolder** — the template ships only the bid-prep scaffolding files; the `proposal/` subfolder (needed for `scripts/render_proposals.py` to produce PDFs / PPTX) is NOT included. A freshly-copied workspace **cannot render** without an additional step. For this workspace, a skeleton `proposal/` was built from the TAMU Wehner pattern via `.scratch/build_leroy_moore_proposal.py`. **Recommendation:** ship a skeleton `proposal/` subfolder inside the template so workspace creation is a single-copy step.
5. **`{{TOKEN}}` placeholders vs `[USER TO FILL]` convention** — the template uses `{{TOKEN}}` syntax for unfilled fields, but active BPC workspaces use the `[USER TO FILL ...]` / `[AWAITING SOURCE DOCS — ...]` convention (which the proposal renderer recognizes and highlights). A workspace freshly copied from the template ships with hundreds of `{{TOKEN}}` markers that the renderer treats as literal text, not as gaps. For this workspace, `.scratch/scrub_leroy_moore_tokens.py` converted each remaining `{{TOKEN}}` to the convention-aligned placeholder. **Recommendation:** add a `firm/_scripts/seed_workspace.py` that takes a template name + project metadata, runs the substitution pass, then converts any remaining `{{TOKEN}}` to the project's convention-aligned placeholder syntax in one go.
6. **`apply_firm_profile.py` is a no-op until past-perf picks are configured** — a fresh workspace doesn't have a `past_project_selection_rules.<slug>.picks` entry in `firm/firm-profile.json`, so Layer 3 of the firm-profile applier produces zero substitutions. For this workspace, the picks rule still needs to be added by hand. **Recommendation:** add a `firm/_scripts/seed_picks_rule.py` that suggests a `picks` list based on the bid's NAICS / scope archetype, then writes a stub into `firm-profile.json` that the user can refine.
7. **Scope-template alignment** — the template's `04-scope-of-work.md` defaults the trade-by-trade table to a generic interior-finishout pattern. For this workspace (3 racquetball courts → teaching space), most of the default trades (lockers, lab gas, tile in wet rooms) are not applicable and should be stripped. **Recommendation:** ship the template with a thinner default trade list and a comment that says "paste the specific scope-template archetype here" rather than carrying every possible trade upfront.

Each numbered item above maps cleanly to a one-line improvement ticket — fold these into Day-3 dispatch.
